title: Overview
description: How this adapter turns the local Amp CLI into an ACP agent, and where each concept is documented.

`acp-go-amp` is a Go ACP adapter around the Amp CLI. It maps Agent Client
Protocol sessions directly onto Amp threads so an ACP host can drive Amp over
stdio or embed the agent in a Go program with `ampacp.Serve`.

The native boundary is the `amp` executable. The adapter does not run an HTTP
service or a native UI; it launches one short-lived `amp threads` process per
prompt and streams the result back as ACP updates.

## What it maps

Each ACP method resolves to a concrete Amp operation:

| ACP method | Amp behavior |
|---|---|
| `session/new` | Creates an Amp thread (`amp threads new`); spends no prompt tokens. |
| `session/prompt` | Continues that thread with `amp threads continue --stream-json`. |
| `session/load` | Replays stored transcript frames for display. |
| `session/resume` | Restores adapter state without replaying history. |
| `session/delete` | Tombstones the local store first, then deletes the Amp thread. |

## What the adapter owns

The adapter is a thin translation layer between ACP and the local `amp` CLI. It
owns the ACP surface:

- ACP session lifecycle and prompt streaming into ACP session updates.
- The local transcript mirror, MCP forwarding, and raw-event diagnostics.
- The intentionally small public contract: the `mode` config option,
  stdio/HTTP MCP setup, raw events, a session-store mirror, and Amp thread
  lifecycle.

## What the adapter does not own

Amp owns authentication, the model backend, and durable thread state on the
server side. The adapter never logs in and never persists credentials, and it
exposes no permission bridging, no adapter-owned model matrix, and no fork
surface.
