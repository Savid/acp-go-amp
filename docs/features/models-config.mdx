description: Configure Amp turns with its select-only mode option, image input, and seeded on-disk native state.

Amp exposes one ACP session config option: `mode`. It is select-only, so a host
presents a fixed value list rather than free text. There is no ACP model option,
because Amp is locked to its hosted backend.

## Config options

| Option | Accepted values | Sent to Amp as |
|---|---|---|
| `mode` | `low`, `medium`, `high` | `-m` |

The adapter passes the selected value straight through. Set it per session
through `_meta.amp.options` (`AmpOptions.Mode`) or update it with
`session/set_config_option`.

## No model option

There is no ACP model config option for Amp. A non-empty `WithDefaultModel`,
`AmpOptions.Model`, or `_meta.amp.options.model` fails session start with the
uniform unsupported error naming field `model`.

## Image input

Send images as standard ACP image blocks carrying embedded base64 data:

```json
{
  "type": "image",
  "data": "iVBORw0KGgoAAA...",
  "mimeType": "image/png"
}
```

The adapter accepts exactly `image/png`, `image/jpeg`, `image/gif`, and
`image/webp`, and only static images. Before starting Amp it decodes the base64
once, checks the declared media type against the bytes, reads dimensions
without rendering pixels, and rejects GIFs with multiple image descriptors,
WebP files with the animation flag, and PNG files carrying an `acTL` chunk.

When an image carries both `data` and `uri`, the embedded data determines the
submitted pixels and the URI is never followed. Image-bearing embedded
resources use the same validation and aggregate budget as image blocks.

Amp routes models on its hosted backend and exposes no authoritative per-model
input modalities, so a structurally valid image is always forwarded and the
native service stays authoritative. The standard `promptCapabilities.image`
capability advertises that this transport exists.

### Limits and input errors

Decoded image bytes are bounded by the configured `ImageLimits`, 6 MiB per
image and 6 MiB across one prompt by default (see
[Go API](/docs/reference/go-api#image-limits)). Two smaller Amp native
envelopes always apply, regardless of configuration:

- at most 5,138,022 decoded bytes per image
- at most 8,000 pixels in either dimension

Crossing a configured byte limit returns `too_large`; crossing a native
envelope returns `native_envelope_exceeded`. Both are `-32602 Invalid params`
errors raised before any native turn starts:

```json
{
  "field": "prompt.image",
  "error": "too_large",
  "index": 1,
  "sizeBytes": 7000000,
  "maxBytes": 6291456
}
```

The full set of input `error` values is `missing_data`, `invalid_base64`,
`invalid_media_type`, `media_type_mismatch`, `animated_not_supported`,
`invalid_dimensions`, `too_large`, and `native_envelope_exceeded`. `index`
counts image-bearing prompt blocks in request order, validation stops at the
first failing block, and an aggregate error names the image that crosses the
total.

## Seed Files

`WithSeedFiles(map[string]string)` maps relative paths to file contents that the
adapter writes into a session's resolved native root before the short-lived
`amp threads` process launches, so the native CLI can read them as its own
on-disk state.

- **Anchor** â€” the per-session isolated `HOME` that `amp` runs with. The adapter
  always creates a fresh isolated `HOME`/XDG root per session under the
  `-scratch-dir`/`WithScratchDir` parent (the system temp directory when that is
  unset), so seeds never land in the operator's real `HOME` or `~/.config/amp`.
- **Placement** â€” seeds are written under `$HOME/<relpath>`. The adapter's
  managed `settings.json` lives under `XDG_CONFIG_HOME` (alongside `HOME`), so
  seeds are written verbatim and can never overwrite the adapter's required
  settings.
- **Confinement** â€” absolute paths, `..` escapes, and empty keys fail closed at
  session start with the uniform unsupported error naming field `seedFiles[...]`.
  Parent directories are created `0o700` and files `0o600`.

<Warning>
Amp does not support bring-your-own-model or custom LLM endpoints â€” it is locked
to its hosted backend. Seed files cannot point Amp at a LiteLLM gateway or any
alternate provider and have no effect on model routing. `WithSeedFiles` is
accepted, but secrets still belong in `WithEnv`, never in a seeded file.
</Warning>
