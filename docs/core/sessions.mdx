---
title: Sessions
description: The mental model for how ACP sessions map onto Amp threads and move through their lifecycle.
---

An ACP session wraps at most one Amp thread. The adapter mints its own UUID
session IDs; the native thread ID (`T-...`) is created by the session's first
prompt turn and recorded in the manifest as `nativeSessionId`. A session that
is never prompted never owns a server-side thread. Durable state lives on the
Amp server side; the adapter keeps only a local mirror for display and
continuity.

## Lifecycle

- `session/new` mints the session id and persists an initial manifest, then
  returns the select config option for `mode`. No Amp process runs, so no
  server-side thread exists until the first prompt.
- The first `session/prompt` runs a thread-less `amp -x` execute: amp creates
  the thread, reports its id on the init frame, and the adapter records it in
  the manifest immediately. Later prompts run `amp threads continue` against
  the recorded thread.
- `session/load` reads the store manifest and transcript, verifies that any
  recorded server-side Amp thread can still be addressed, recreates the
  adapter session with a fresh isolated native home, and replays stored frames
  to the client.
- `session/resume` performs the same continuability check without replaying
  history.
- `session/delete` writes the store tombstone first, closes any active local
  session, then calls `amp threads delete` when the manifest records a native
  thread; a session with no recorded thread deletes locally only. A native
  delete failure is returned after the local session is already hidden from
  list/load/resume, and cleanup is retried by later list/load/resume/delete
  calls.
- `session/close` stops accepting new prompts for the local session and removes
  its isolated native home. It does not delete the server-side Amp thread.

## Server-side state and the local mirror

Because state is server-side, the adapter cannot rebuild a thread from local
history. The transcript mirror and canonical image records exist to replay
display updates on load — not to recreate native state. Continuation always
requires the live server-side Amp thread and `AMP_API_KEY`; if the thread is
gone, load still replays local history but the next prompt returns a terminal
`native_state_missing` error. See
[Session store](/docs/features/session-store) for the durability boundary.

## Mirror commit ordering

Only a clean `end_turn` publishes its frames to the store; cancelled or errored
turns are never mirrored locally, even though the server-side thread advances
regardless. If a store write fails after a completed turn, the adapter keeps the
exact frames in memory and blocks the next prompt with a `mirror_unsynced`
error until the mirror is reconciled, so the local record never silently drifts
from what Amp produced.
