---
title: Prompt Streaming
description: How one short-lived Amp process per prompt turns stream-JSON frames into ACP updates and results.
---

Each ACP prompt starts exactly one short-lived native process and streams its
output back as ACP `session/update` notifications, ending with a single
terminal result.

## The prompt process

```text
amp threads continue <thread> --stream-json --stream-json-input -x
```

The adapter always adds `--no-ide`, `--no-color`, `--no-notifications`, and an
isolated `--settings-file`. MCP configuration is supplied with `--mcp-config`
when the ACP session includes stdio or HTTP MCP servers.

## Frame mapping

Amp stream-JSON frames map to ACP updates:

- user text frames become user message chunks
- assistant text frames become agent message chunks
- tool-use frames become tool-call updates
- usage data becomes ACP `usage_update`
- final result usage is copied to the prompt response

Main-agent message chunks carry a wrapper-derived UUID in `messageId` and
`_meta.amp.messageId`; the successful prompt response repeats the terminal
main-agent value. Amp exposes no native message/turn id, so the adapter anchors
the UUID to the exact raw frame and its durable transcript position. This keeps
load replay stable without modifying native mirror bytes.

The `usage_update` `size` field carries Amp's native `usage.max_tokens`
verbatim (a model-scale value Amp reports), or `0` when absent.

No slash commands are advertised. Text like `/review` remains ordinary prompt
text and is sent to Amp unchanged.

## Cancel and stop reasons

`session/cancel` sends an interrupt to the active prompt boundary. On Linux,
each turn runs below a dedicated child subreaper. It adopts and terminates the
whole descendant tree even when a native tool escapes the original process
group with `setsid(2)`. On Windows, the turn is assigned to a Job Object and
the adapter waits for its active-process count to reach zero. If graceful
interruption exceeds the configured native cancel timeout, the entire boundary
is terminated.

Prompt settlement is fenced by that containment proof. A successful turn, a
cancelled turn, and a turn-timeout failure are not returned from
`session/prompt` until every process in the turn boundary has exited. If the
adapter cannot prove quiescence, it returns an error matching
`ampacp.ErrProcessTreeUnproven` instead of the otherwise terminal result and
retains resources that the unproven tree may still reach.

On platforms without the Linux subreaper or Windows Job Object backend,
session startup and native operations fail closed rather than launching a turn
with partial containment.

A successful prompt returns exactly one terminal result, and this adapter
emits only two of the ACP stop reasons: `end_turn` for a completed turn and
`cancelled` for a cancelled one. The other reasons in the ACP enum
(`max_tokens`, `max_turn_requests`, `refusal`) exist but this adapter never
produces them, so a host must not wait for them.

## Turn failures and timeouts

A native turn failure is never reported as a stop reason. It terminates
`session/prompt` with a JSON-RPC error (`-32603`) carrying:

```json
{
  "error": "amp_turn_failed",
  "cause": "process_exit" | "transport" | "provider" | "timeout",
  "message": "<real native cause>"
}
```

`message` always carries the real native cause (exit status and stderr tail for
a process exit, the decode/read error for a transport fault, or `result.error`
falling back to `result.result` for a provider error) — never a fixed
placeholder. The session stays addressable and retriable after such a failure; a
follow-up `session/prompt` re-drives the turn.

<Note>
`WithTurnTimeout(d)` sets an optional per-turn deadline (default `0`, no
deadline). On expiry the adapter aborts the native turn and returns the failure
error with `cause: "timeout"` — a timeout is a failure, not a cancellation, so
it never produces a `cancelled` stop reason.
</Note>
