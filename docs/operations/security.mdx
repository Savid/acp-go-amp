---
title: Security
description: How per-session isolation and strict secret handling keep credentials out of durable state.
---

The adapter isolates each session's native state and keeps credentials out of
anything durable. Operators can reason about the blast radius of a single
session and about exactly what the session store may contain.

## Per-session isolation

The adapter creates an isolated native home for each ACP session. Native `HOME`,
`XDG_CONFIG_HOME`, `XDG_CACHE_HOME`, `XDG_DATA_HOME`, and `XDG_STATE_HOME` point
inside that directory, and the Amp settings file starts as an empty JSON object
inside the isolated config tree before being passed with `--settings-file`. The
directory is removed on close/delete, so one session cannot read another's
native state or the operator's real `~/.config/amp`.

## Secret handling

Auth material is live process state only:

| Value | Supplied via | Persisted? |
|---|---|---|
| `AMP_API_KEY` | `WithEnv` or the process environment | Never |
| `AMP_URL` | `WithEnv` or the process environment | Never |

Neither value is written to the settings file or the `SessionStore`. The store
contains only the manifest and captured stream-JSON frames — no settings, auth
files, API keys, or other secrets.

<Warning>
Seed files (`WithSeedFiles` / `-seed-file`) are written verbatim into the
isolated native home. Never seed secrets through them; put credentials in
`WithEnv` so they stay in live process state and out of durable files.
</Warning>

## Stream isolation

Native Amp subprocesses never inherit ACP stdout. The adapter captures native
stdout and stderr through dedicated pipes so JSON-RPC output stays clean and
native diagnostics cannot leak into the protocol stream.

## Native-process containment

Linux starts every Amp command — version/readiness probes, thread lifecycle
commands, and prompts — below an authoritative child subreaper. Descendants
that detach with `setsid(2)` are reparented, terminated, and reaped until
`wait4(2)` reports `ECHILD`. Windows assigns every command tree to an
authoritative Job Object and waits for its active-process count to reach zero.
Cancellation, timeout, and successful completion all cross the same selected
boundary. An incomplete boundary surfaces
`ampacp.ErrProcessContainmentIncomplete`; reachable native-root and scratch
resources are retained, and the parent-side wait remains bounded.

Darwin fails closed unless the operator explicitly supplies
`-darwin-best-effort-containment` or
`WithDarwinBestEffortContainment`. The opt-in creates a fresh real writable
generation root for every native launch, starts the direct child as its own
process-group leader, reaps that child exactly once, sends group `SIGTERM`,
escalates to `SIGKILL` after 500 ms, and polls the original group for one fixed
five-second total deadline. It records `containment=best_effort` and emits an
unconditional warning.

This Darwin boundary does not contain descendants that call `setsid(2)` or
otherwise escape the original group. Such processes can become zombies or
continue running. Inherited runtime markers are correlation, not ownership;
they can be scrubbed, and an empty scan is not evidence of absence. A negative
PID signal also targets a numeric PGID rather than an identity-bound handle, so
PGID reuse can cause collateral signalling. Native-root permits do not bound
escaped provider work, and provider-descendant snapshot metrics are suppressed
instead of publishing a misleading count or zero.

Operators can inspect retained records without mutation and explicitly clean a
selected runtime:

```sh
acp-go-amp containment diagnose -scratch-dir /path/to/scratch
acp-go-amp containment cleanup -scratch-dir /path/to/scratch -runtime-id <id> -force
```

Cleanup signals only individually revalidated same-uid marker-correlated PIDs;
it never treats a recorded PID or PGID as signal authority. Ambiguity retains
the generation root and returns a non-zero status. Its fixed five-second budget
starts before the first process enumeration. PID revalidation and signalling
are separate operations: reuse in that gap can still signal unrelated same-uid
work. These commands remain best-effort diagnostics, not a claim that
marker-scrubbing escapees are absent.
